# 戒烟网页设计说明

日期：2026-04-28

## 1. 目标

构建一个移动端优先的戒烟网页，服务单用户自用场景，支持跨设备同步。网页围绕正在服用酒石酸伐尼克兰进行戒烟的使用背景设计，但不提供剂量建议、疗程判断或医疗诊断。

v1 产品流程采用“先开始服药、后设定戒烟日”的路径：允许在前期继续抽烟并记录，待进入合适窗口后设定戒烟日，再进入正式戒烟阶段。该流程用于贴合当前使用场景，不覆盖说明书中的全部用法路径。

产品目标：

- 提供每日服药打卡能力
- 提供前期抽烟记录与减烟观察能力
- 在烟瘾出现时提供快速、低负担的应对流程
- 记录副作用、情绪、复吸诱因等事件，支持后续回顾
- 通过无烟时长、节省金额、恢复节点等内容提供正反馈

非目标：

- 不做复杂账号体系
- 不做多用户支持
- 不做自动医疗判断
- 不把伐尼克兰固定疗程硬编码进系统

## 2. 用户与约束

用户为单一使用者，自用为主，要求：

- 移动端体验优先
- 页面具有设计感和交互感
- 数据可跨设备同步
- 可接受轻量后端
- 数据存储在 SQLite 中

工作边界：

- 允许轻量服务端和 SQLite
- 浏览器本地存储不能作为主数据源
- 后端以单用户场景优化，不引入复杂认证系统
- v1 不内建完整登录系统，默认部署前提是私有环境，或由反向代理 / 基础认证提供访问保护
- v1 以“开始服药后在第 8 至 35 天内设定戒烟日”的产品路径为主

## 3. 设计方向

采用暖色、陪伴感、恢复进度导向的视觉方向。

设计特征：

- 暖色渐变背景和半透明卡片
- 首页强调“先把今天稳住”
- 正反馈用语和恢复感视觉层级强于数据仪表盘感
- 交互风格接近轻量 App，而不是营销展示页

## 4. 信息架构

底部导航固定为三个主页面：

- `今日`
- `记录`
- `进展`

产品分为两个阶段：

1. `预备期`
   - 已开始服药
   - 尚未进入正式戒烟日
   - 允许继续抽烟
   - 重点是服药打卡、抽烟记录、诱因观察、逐步减烟
2. `戒烟期`
   - 已到达或超过戒烟日
   - 继续维持药物服用
   - 重点转为烟瘾应对、复吸记录、无烟进展

核心使用路径：

1. 用户打开网页进入 `今日`
2. 先确认今日服药状态
3. 在预备期记录当天抽烟情况，必要时设置戒烟日
4. 在戒烟期若有烟瘾，立即进入短流程应对
5. 若有不适或事件，快速记录到时间线
6. 在 `进展` 查看减烟或戒烟成果

## 5. 页面设计

### 5.1 今日

定位：主操作页，第一屏只服务于当天最重要的动作。

核心模块：

- 今日状态头部
  - 当前日期
  - 预备期：治疗第几天 / 距戒烟日还有几天
  - 戒烟期：无烟第几天
  - 简短状态文案
- 今日服药卡
  - 根据用户配置的服药时段展示状态
  - 显示已服用、待服用、漏记等状态
  - 支持快速打卡与备注
- 阶段核心卡
  - 预备期：今日已抽支数、对比基线、快速记一支、设置戒烟日
  - 戒烟期：当前烟瘾强度、一键进入抗烟瘾流程
- 快速记录入口
  - 副作用
  - 情绪波动
  - 复吸诱因
  - 烟瘾发作

设计原则：

- 第一屏根据阶段切换，但都不承载复杂统计
- 一个主按钮优先于多个平级操作
- 让用户在几秒内完成最重要动作

预备期补充模块：

- `今日已抽`
  - 显示今天累计抽烟支数
  - 对比基线日均支数
  - 提供“快速记一支”入口
- `戒烟日设置`
  - 当治疗日数达到第 8 天且尚未设定戒烟日时，显著提示设置戒烟日
  - 戒烟日可选范围限制在治疗第 8 至 35 天

戒烟期补充模块：

- `当前烟瘾`
  - 用 1 个主按钮进入短流程应对
- `未完成回访`
  - 若存在 10 分钟回访未完成，优先在首页展示

### 5.2 记录

定位：统一时间线，用于承接服药之外的事件与观察。

时间线数据来源为 `record_entries` 与 `smoking_logs` 的合并视图。

核心模块：

- 时间线列表
  - 按时间倒序展示事件
  - 每条记录显示类型、时间、强度、是否缓解、备注摘要
- 快速新增入口
  - 使用底部抽屉或弹层，不跳转整页
- 筛选能力
  - 按类型筛选：抽烟、副作用、烟瘾、情绪、诱因、复吸

设计原则：

- 低负担输入优先于长文本记录
- 结构化字段优先于纯自由文本
- 保留补充备注，但不是必填

### 5.3 进展

定位：强化反馈与回顾页，不打断日常主操作路径。

核心模块：

- 预备期
  - 近几日抽烟趋势
  - 相对基线的减烟幅度
  - 服药依从性
- 戒烟期
  - 无烟时长
  - 少抽支数
  - 节省金额
  - 身体恢复节点
  - 本周常见诱因
  - 最有效的应对动作

设计原则：

- 强化“已经做到的事”
- 用回顾帮助发现规律
- 信息密度低于工具后台，保持陪伴感
- 在预备期强化“减烟进展”，在戒烟期强化“无烟进展”

## 6. 关键交互

### 6.1 服药打卡

逻辑：

- 用户先在设置中配置自己的服药时段
- `今日` 页按时段展示今日状态
- 点按后写入一条服药记录
- 可附带简短备注，例如“轻微恶心”“饭后服用”

约束：

- 不预置伐尼克兰剂量规则
- 不对异常情况给出剂量建议

### 6.2 戒烟日设置

逻辑：

- 首次使用时记录治疗开始日期
- 用户可暂不立即设定戒烟日
- 当治疗日数进入第 8 天后，系统提示用户设定戒烟日
- v1 将戒烟日可选范围限制在治疗第 8 至 35 天
- 到达戒烟日后，产品状态自动切换为 `戒烟期`

展示规则：

- 预备期首页显示“距戒烟日还有几天”或“请设置戒烟日”
- 戒烟期首页显示“无烟第几天”

### 6.3 前期抽烟记录

预备期允许正常抽烟，但要求结构化记录，以便观察减烟趋势与诱因。

交互：

- `今日` 页提供“快速记一支”按钮
- 每次记录至少包含发生时间和支数，默认支数为 1
- 可选补充场景、诱因、备注
- 支持批量补记，例如“今天上午抽了 3 支”

目的：

- 跟踪从基线到戒烟日前的减烟变化
- 为后续 `进展` 页提供趋势和对比
- 帮助识别高频场景与诱因

### 6.4 烟瘾应对流程

流程控制在短路径内，避免复杂问答。

建议流程：

1. 选择当前烟瘾强度
2. 选择诱因或场景
3. 选择一个应对动作
4. 生成一条烟瘾记录
5. 稍后提供“是否扛过去”的回访入口

目标：

- 让用户在烟瘾出现时有即时动作
- 将数据结构化，方便后续回顾

v1 明确约束：

- 不做系统级通知或推送
- 每次烟瘾记录创建后，同时创建一个待回访状态，目标时间为记录后 10 分钟
- 如果用户当时仍停留在应用中，则在 `今日` 页内联出现回访卡片
- 如果用户离开后再次打开应用，则在下一次进入 `今日` 页时优先显示未完成回访

### 6.5 快速记录

新增记录时，根据类型动态展示字段：

- 抽烟记录：时间、支数、场景、备注
- 副作用：表现、严重度、备注
- 烟瘾发作：强度、诱因、应对动作、是否缓解
- 情绪波动：情绪类型、强度、备注
- 复吸诱因：场景标签、备注
- 复吸：时间、支数、原因、备注

### 6.6 回顾

`进展` 页按周聚合：

- 预备期：每日抽烟支数、相对基线变化、高发抽烟场景
- 高发诱因
- 高发时段
- 最有效应对动作

该页的目标不是做复杂分析，而是提供可执行的自我观察结果。

## 7. 技术架构

采用单仓全栈 Web 应用：

- 前端：移动端优先的 React/Next.js 应用
- 后端：同项目内的轻量 API 路由
- 数据库：SQLite
- ORM：Prisma 或 Drizzle，二选一即可

推荐采用 Next.js 方案，原因：

- 页面、API、样式和移动端适配可在一个项目中完成
- 适合快速构建单用户工具型产品
- 后续增加聚合查询或表单交互成本较低

## 8. 数据模型

### 8.1 settings

单用户配置表，用于存储基础参数与服药计划。

字段：

- `id`
- `treatment_start_date`
- `target_quit_date`
- `baseline_cigarettes_per_day`
- `cigarette_price_per_pack`
- `cigarettes_per_pack`
- `timezone`
- `medication_name`
- `medication_schedule_json`

### 8.2 medication_logs

记录每次服药行为。

字段：

- `id`
- `scheduled_for_date`
- `scheduled_slot`
- `taken_at`
- `status`
- `note`
- `created_at`
- `updated_at`

### 8.3 record_entries

统一记录时间线事件。

字段：

- `id`
- `occurred_at`
- `kind`
- `severity`
- `trigger_tag`
- `action_tag`
- `resolved`
- `note`
- `created_at`
- `updated_at`

`kind` 初始取值：

- `craving`
- `side_effect`
- `mood`
- `trigger`
- `slip`

### 8.4 smoking_logs

记录抽烟行为，重点服务预备期，也可在戒烟期承接复吸后的数量记录。

字段：

- `id`
- `smoked_at`
- `count`
- `context_tag`
- `trigger_tag`
- `note`
- `created_at`
- `updated_at`

说明：

- `count` 支持一次记录多支，便于补记
- 是否处于预备期或戒烟期由 `smoked_at` 与 `target_quit_date` 推导，不单独存冗余阶段字段

### 8.5 craving_followups

记录烟瘾应对后的待回访状态。

字段：

- `id`
- `craving_record_id`
- `followup_due_at`
- `followup_completed_at`
- `outcome`
- `note`
- `created_at`
- `updated_at`

### 8.6 daily_stats

初期不单独持久化，优先通过查询动态聚合。如首页性能或复杂度不足，再增加缓存或物化表。

## 9. 后端边界

后端只负责以下职责：

- 单用户数据读写
- 今日视图所需状态聚合
- 进展页所需统计聚合
- 基础配置读取与更新
- 阶段判定与首页状态切换

不负责：

- 医疗判断
- 剂量推荐
- 外部消息推送系统
- 多用户权限管理
- 独立认证中心

## 10. 异常处理

- 同一时段重复服药打卡时，提示改为补充备注或修正时间
- 戒烟日未设置且治疗已进入第 8 天后，首页持续提示设置戒烟日
- 戒烟日前后的指标切换要明确，避免将预备期误显示为“无烟失败”
- API 写入失败时，前端显示明确失败状态与重试入口
- 记录输入允许最小必填字段，不强制长备注
- 连续出现明显不适时，只提供通用提示：联系医生或查看说明书

## 11. 测试策略

单元测试：

- 无烟天数计算
- 预备期 / 戒烟期状态切换判定
- 节省金额计算
- 今日服药状态判断
- 聚合统计函数

API 测试：

- 服药打卡
- 戒烟日设置合法范围校验
- 重复时段冲突处理
- 新增抽烟记录
- 新增记录
- 读取时间线
- 读取进展聚合

交互测试：

- `今日` 页服药打卡流程
- `今日` 页预备期快速记一支
- `今日` 页从预备期切换到戒烟期的显示变化
- 烟瘾应对流程
- `记录` 页抽屉新增记录

端到端测试：

- 首次初始化
- 预备期完整使用路径
- 设置戒烟日并跨日进入戒烟期
- 单日完整使用路径
- 页面刷新后数据一致

## 12. 实现原则

- 先完成最小可用闭环，再补充装饰性交互
- 视觉设计服从移动端操作效率
- 首页优先保证“服药打卡 + 当前阶段最重要动作”路径顺畅
- 记录结构优先支持回顾，而不是追求一次性填很多内容
- 所有医疗相关表述保持保守，不越界
